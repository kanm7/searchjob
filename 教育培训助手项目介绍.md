# 教育培训助手项目介绍

这是一个基于SpringAI框架的智能教育系统，通过AI Agent技术，构建了一个能够：

理解学习者的需求和水平，并且能根据此来规划个性化的学习路径，然后生成针对性的学习内容，并且可以评估知识掌握程度，然后灵活调整学习策略。

关于代码，只是少部分涉及SpringAI框架的内容，大部分都是自己手动实现的。

手写了哪些部分：

像

- 手写意图路由和核心编排
- 手写三层对话记忆压缩策略
- 手写skill/command双模式路由与工具统一管理机制
- Rag检索系统
- 手写subAgent记忆隔离与生命周期管理

#### 项目运转的主要流程是：

用户输入需求，然后再大模型回答之前呢，会先进行意图识别，然后针对性的选择工具，而不是将所有工具都发给大模型，之后就是构建prompt，再调用大模型，如果需要的话，执行工具，再返回结果，大模型在进行总结并且回复用户。整体流程就是这样的。

## 创新点

#### 一、**动态工具路由与意图识别**

传统做法是将所有工具都传给LLM，但是这样会导致：

- Prompt过长，消耗大量token
- LLM选择困难，导致准确率下降
- 响应速率相应也会变慢

"动态路由的核心思想是：先识别用户意图，再根据意图按需加载相关工具，避免把全部工具都传给大模型。这样可以减少部分 Token 消耗，提升响应速度和工具调用准确率。"

我的方案是，只传递相关的工具模块给大模型。代码实际运行，这个效果是立竿见影，token消耗也少了一半，响应速度也快了很多。如何定义这个相关的工具呢，我们是建立了意图到功能的一个映射关系，然后提供给大模型一段few-shot这种参考，然后让大模型根据用户输入分析需要用到哪些工具，这样其实是要比上来就把工具全提供给大模型效果要好的。

##### 意图识别的准确率怎么样？识别错了怎么办？

准确率：约 87%（通过 200 条测试集验证）

##### 这样做能节省多少 Token？有数据支撑吗？

数据对比（200 次对话测试）：

无优化（全量工具）：
- 平均 Token/次：4200
- 工具描述占比：约 60%

意图识别 + 动态路由：
- 平均 Token/次：2100
- 工具描述占比：约 20-30%
- 节省：50%

具体场景：
- 天气查询：从 4200 → 800（节省 81%）
- 知识检索：从 4200 → 1500（节省 64%）
- 代码审查：从 4200 → 1800（节省 57%）

##### 如果我要新增一个工具或意图，需要改哪些地方？

首先是，新增工具，要实现innertool接口，定义工具；其次就是新增意图，添加意图和功能的映射关系

#### 二、三层对话记忆压缩策略

长对话会导致token溢出，成本变高，响应变慢，我是利用了记忆的三层压缩策略：

- 摘要压缩
  - 我是这样约定的，当历史消息队列的长度超过60条，我就只保留最近的20条消息，特殊的像大模型调用工具这种，需要特判一下，大模型需要把调用工具和工具回复一起看才能看懂，不然大模型就会看不懂。时间较长的消息呢就和旧摘要进行压缩形成新的摘要。
- Assistant裁剪
  - 只保留最近的5条assistant回复，因为LLM回复的内容一般是比较长的，消耗token比较多，但是一般说来，用户的问题是要比大模型的回复有价值的，即使丢弃了早期的大模型的回复，那我还们还保留早期的问题以及摘要，那么也不会损失有价值的内容，这样的话比较节省token。
- 滑动窗口兜底
  - 硬性限制最大消息数量的。

所以构建的prompt，就是历史摘要+最近的消息+用户输入的新问题，然后配合上相对应的工具。

#### 三、多路召回+RRF融合+Reranker重排的RAG系统

##### 关于常见分块策略：

- 固定大小分块，会破坏上下文语义完整性，可以设置块与块之间的重叠程度
- 递归字符文本分块，按照字符的优先级进行切分
- 语义分块，计算相邻句子的语义相似度，如果相似度高，就放到一个块中
- 基于文档结构的分块，像markdown、html这种有结构的文档使用
- 元数据增强分块，就是给块增加一个属性信息，大模型会更好的理解文档的内容

单一召回的局限：

- 向量检索，擅长语义相似，但可能会漏掉关键词精确匹配
- 关键词检索，但它不懂语义

我的方案是：三路召回+RRF融合

向量检索，我是通过余弦相似度进行检索的

Bm25，通过计算关键词的词频TF和逆文档频率IDF，来评估相关性的。

查询改写，运用大模型，将用户输入的描述改写成其他几种表达，分别检索

在将三路检索到的文档经过RRF融合算法排序，得到最终排名。最后再用专门的LLMReranker模型进行精确排序，选出最相关的三个内容，拼接到Prompt中丢给大模型。这个模型我用的是智谱的rerank模型，如果调用失败了，会使用RRF融合后的排序了。

#### 四、模块化可插拔工具系统

实现了一个统一的工具接口，工具想要被大模型使用，都得先实现tool接口，然后spring在启动的时候，会自动扫描所有实现了tool接口的工具类，然后加载到IOC容器中。并且，后续如果扩展新的工具，也不用改太多代码。

#### 五、MCP协议双向支持

我是同时实现了mcp server和mcp client两种。

##### 作为client

- 通过在前端页面上可以手动添加mcpserver，然后自动保存到本地，如果后续系统重启，mcp配置也不会丢失。优先使用的是Streamable http，失败的话回退sse。用try-catch实现优雅降级。
- 我是连接了钉钉的mcpserver，它提供了像发送消息、查询日程、创建群组等工具。
  - 为什么选钉钉呢？
    - 在这种教育背景下，我了解到，现在很多学校和培训机构，都通过使用钉钉来发布作业，签到等等，考虑到这，所以我就用的钉钉

##### 作为server

- 对外，我是暴露了教育系统的能力，其他的AI应用可以通过mcp协议调用服务

#### 七、实际应用案例

##### 案例1：初学者入门

**场景**："我想学习 Java，完全零基础，每天2小时，希望6个月后找工作"

**系统响应**：

1. 识别意图：EDUCATION
2. 调用 LearningPathTool 生成6个月学习路径
3. 调用 learning_plan_generator 生成首月详细计划
4. 每周自动生成练习题
5. 定期评估进展并调整计划

**价值**：相当于拥有一个私人学习顾问，成本仅为传统培训的 1/10。

##### 案例2：知识巩固

**场景**："刚学完 Java虚拟机相关知识，想测试掌握程度"

**系统响应**：

1. 生成10道测试题（混合题型）
2. 用户作答后自动评估
3. 识别薄弱环节
4. 生成针对性总结和建议

关于秒杀业务，是这个项目中的核心业务场景，这是一个典型的高并发场景。我采用四层防护架构来应对瞬时高并发请求，分别是：网关限流-》redis预减库存-》使用消息队列异步创建订单-》数据库最终写入订单，来同时保证数据的一致性和可靠性。下面我详细说明一下：

首先，在网关层使用全局限流+用户限流的策略，采用令牌桶算法限流，防止服务被大量请求打崩。

然后，秒杀的核心难点是防止超卖，如果先查库存再扣减，就会出现比如，现在库存只剩下一件，有A、B用户分别尝试下单，那A如果先查库存，发现库存还有，此时发生了线程切换，B也查到还有一件库存，那么A、B都认为库存是充足的，那么A、B都会下单成功，就发生了超卖。解决方法就是使用redis+lua脚本的方式，原子性的完成库存查询和库存扣减，然后把利用mq创建订单。

那如果不用mq，mysql是扛不住瞬时高并发的写请求的，改用mq之后，交易服务就可以按照自己的处理能力来处理消息。创建订单之后，使用延迟队列，来处理最终的付款，如果到时间后还没付款，就取消订单，并且要回补库存。

### Redis 扣减库存成功，但发送到 RabbitMQ 的消息在网络传输中丢失了，导致数据库没创建订单，用户钱扣了但没货，你怎么解决这个数据一致性问题？这个问题怎么回答？

#### 这其实是两个问题，第一个问题是redis扣减成功，但是发送到mq失败了

这里我是通过记录日志的方式，因为这里的话日志表相较于订单表，很轻量，数据库压力很小。那么在redis发送成功后，更新日志状态，发送mq消息成功后，更新日志状态。然后设置定时任务，不断扫描日志表，针对这些异常日志，重新操作。

#### 第二个问题是用户支付成功，但是支付订单失败了

回调接口里，通过给mq发送订单支付成功消息，这样，即使数据库修改订单状态失败，消息也不会丢失，会进行重试，或者把消息丢进死信队列中，作为异常状态订单。系统会定时扫描状态异常订单，同时也会去定时扫描说长时间未支付的，然后主动查询支付宝订单状态，再去补充更新数据库；如果支付宝回调失败，支付平台都会多次重试，直到返回success，所以，支付回掉接口必须幂等且可重复执行。什么叫幂等，比如说，支付宝连续回调5次，系统不能加5次余额，发5次货，必须做到已经处理过的，直接返回成功。

### 关于保证mq可靠性

使用生产者的publish confirm和publish return机制，确保消息一定到达mq上，同时开启mq的持久化，保证消息宕机不会丢失。同时消费者端开始手动ack机制，确保一定写到数据库才确认。同时呢，利用死信队列捕获处理失败的消息。

### 关于订单服务的幂等性

mq可能重复投递消息，比如消费者宕机后重试，那么后端就需要保证同一个订单不会创建多次，我是采用的分布式锁+数据库唯一索引来保证订单的幂等性的。

### 订单超时自动取消+库存回补

用户下单成功之后，万一不付款，不能让用户一直占用库存，就需要自动取消订单并且恢复库存。这里我是采用的ttl+死信队列实现延迟消息。就是设置消息在队列中最多存储30分钟，超过之后，就可以被消费者捕获来处理了

### 关于分布式事务Seata

Seata的三个核心角色，分时是：

- TC，负责管理全局事务，它是一个独立运行的服务器
- TM，事务的发起者，在秒杀业务里面，就是在秒杀的那个方法上加一个@GlobalTransaction注解，就相当于它会告诉TC要开启一个全局事务
- RM，相当于每个参与事务的微服务，它们负责管理本地事务

就举个例子，TM告诉TC要开启全局事务，TC就会给分配一个全局唯一的事务id，然后，TM在远程调用其他微服务的时候，就会把这个事务ID传递下去，RM发现有事务ID，就知道自己正在参与一个全局事务，于是就会向TC注册分支事务，最后TC负责汇总所有RM的汇报，如果大家都说没问题，就发通知全局提交；只要有一个报错了，TC就通知全局回滚。

#### **既然跨了多个数据库，Seata 是怎么在失败时把数据『变』回去的呢？**”

我使用的是Seata的AT模式，简单来说就是，RM在修改数据库之前，会自动生成一份数据快照，存到undo_log中，如果需要回滚，就可以恢复修改之前的数据了。

#### 关于远程调用部分

项目中原本使用的是 Spring Cloud Alibaba 体系的 OpenFeign。改为 gRPC 后，gRPC 基于 HTTP/2 和 Protobuf，相比于 OpenFeign（基于 HTTP/1.1 和 JSON），序列化体积更小，解析速度更快，非常适合作为微服务内部的高性能 RPC 框架。

#### 如何区分普通商品和秒杀服务呢？

在项目中，我是通过建立独立的“秒杀活动表/秒杀商品表”来进行绑定的，在业务实现上，通过判断当前商品是否属于秒杀商品、是否在秒杀活动时间范围内，如果是秒杀商品，会走秒杀链路；正常商品就走正常的下单链路。

如果面试官问，为什么不在原表上添加一个是否是秒杀商品的字段，可以这样回答：

秒杀是一种高频词、高并发的活动，如果直接在原表上添加is_secskill字段，可能会导致两个严重问题吧：

- 缓存及其难做：商品基础信息的缓存过期时间比较长，而秒杀商品的库存信息是实时变化的，放在一起会导致缓存一致性难以维护。
- 锁竞争严重：秒杀扣减库存会产生很高的并发，如果扣减的是普通商品表的库存，会导致整张表甚至是整个库的性能收到影响，拖垮数据库的性能，影响其他正常商品的交易。

#### https和tls加密过程

https使用的是加密传输，保证了安全。https需要从CA申请TLS证书。

TLS协议的机制是这样的：

浏览器和服务器在进行数据传输之前，需要先进行TLS握手，主要就是商量使用的加密方法、以及确认对方身份。常见的加密方式有两种，对称加密和非对称加密。但是现在的网络传输都采用混合加密机制，就是在TLS握手阶段，服务器把公钥发给浏览器，浏览器生成一个加密钥匙，用公钥加密后，在传给服务器，这样只有有私钥的服务器才能解密获取这把加密钥匙，后续在进行数据传输时，双方都用这把加密钥匙进行加密就可以。

#### TCP三次握手：

TCP三次握手是建立可靠连接的过程。首先，客户端向服务端发送SYN请求报文请求建立连接，服务端收到并回复AcK+SYN报文表示同意建立连接并且同步自己的初始序列号。然后客户端返回Ack报文确认连接。三次握手的核心目的就是为了：同步双方的初始序列号以及确认双方都具备完整的接收和发送数据的能力。

#### 为什么两次握手不行？

三次握手是服务器为了确保客户端真的接收到自己的消息后才正式分配资源

##### Tcp三次握手最后一个ack包没有送到就开始传输数据包，会发生什么？

服务端不会接收数据包，并且会返回给客户端一个异常包。

##### 关于Tcp可靠传输

通过序列号和ack机制确保顺序，并且发送方没发送一个报文，就启动一个重传定时器，若在规定时间内未收到ack报文，就重传。但是超时重传太慢了，当发送方连续收到3个重复的ack（即表示对之前报文的确认，也携带了下个报文的序号信息，例如ack=1000），就认为1000号字节丢失，立即重传，不等超时。同时也具备流量控制，通过滑动窗口限制发送速率，使发送速度不超过接收方的处理能力。

##### Tcp拥塞控制

拥塞控制是为了防止网络过载，基于拥塞窗口，实际发送窗口大小=min（拥塞窗口，滑动窗口）。还有一个慢开始门限。拥塞控制经历多个阶段，分别是：

- 慢开始，当拥塞窗口 《 慢开始门限，每次发送完都让拥塞窗口翻倍
- 拥塞避免，当拥塞窗口》=慢开始门限了，每次发送完都让拥塞窗口+1
- 快重传，快恢复：当网络拥塞发生丢包，当发送方收到连续的三个ack报文，发送端会将慢开始门限设置为当前拥塞窗口的一半，并且也让拥塞窗口设置为减半的值，然后再开始执行拥塞避免算法

#### TCP四次挥手：

tcp四次挥手是释放连接的过程。首先，主动方发送FIN报文表示不再发送数据，但是可以接收数据；被动方回复Ack确认，此时连接进入半连接状态，被动方仍然可以发送未传完的数据。当被动方发送完数据后，会发送自己的FIN报文。最后，主动方回传Ack确认，并在等待2MSL后彻底关闭连接，以确保最后一个Ack能够送达，并防止就链接报文干扰后续的新连接。

```
客户端                         服务器
ESTABLISHED                  ESTABLISHED
   |                               |
   | (1) FIN                      |
   |----------------------------->|
   |                         (2) ACK
   |<-----------------------------|
FIN_WAIT_1                     CLOSE_WAIT
   |                               |
   |                            (完成数据发送)
   |                         (3) FIN
   |<-----------------------------|
FIN_WAIT_2                     LAST_ACK
   |                               |
   | (4) ACK                      |
   |----------------------------->|
TIME_WAIT                          |
   |                               |
   | (2MSL 后)                     |
   V                               V
CLOSED                         CLOSED
```



##### 为什么不是三次挥手，像三次握手一样？

握手时，任何一方都可以同时发送syn+ack合并为一个报文。但挥手时，被动方收到fin后，可能还有未发完的数据，不能立即关闭自己的发送通道，所以第2步和第3步不能合并，需要两次独立的报文。但是，如果被动方在收到fin后，没有要发的数据了，理论上可以将ack+fin合并为一个报文，只不过tcp协议并没有强制要求合并。

##### TimeWait状态的作用？

主动方在收到被动方的fin报文后，需要发送ack报文给被动方，那么这个过程，ack报文有可能丢失，那么当被动方没有在规定的时间内收到ack报文，就会触发超时重传fin报文，所以需要有timewait状态，来防止ack报文丢失。如果计时器超过了2MSL，就表示被动方已经接收到ack报文了，主动方关闭连接就可以了。

##### tcp为什么是可靠传输，而udp不是？

tcp通过序列号、ack、超时重传、流量控制、拥塞控制等机制确保数据可靠有序到达。

udp只是把数据发过去，不关系是否到达。

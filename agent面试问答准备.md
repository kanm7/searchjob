### 2. 向量嵌入用的哪个模型?为什么要用这个?这个向量嵌入过程是什么?

**你的项目实现:**
- 使用 Spring AI 的 `EmbeddingModel` 接口
- 配置文件中使用的是通义千文模型

**面试回答要点:**

1. **用的什么模型**:
   "我们使用的是通义千文的 embedding 模型(通过 Spring AI 的 EmbeddingModel 接口调用)"

2. **为什么用这个**:
   - 中文支持好,对于中文文档的语义理解更准确
   - 与我们的技术栈兼容(Spring AI 原生支持)
   - 性价比高,调用成本相对较低
   - 2048维的向量维度,在精度和性能之间取得平衡

3. **向量嵌入过程**:
   
   ```
   第一步:文本预处理
   - 清洗文本(去除特殊字符、规范化)
   - 分块(500字符/块,50字符重叠)
   
   第二步:调用 Embedding API
   - 将文本块发送到 embedding 模型
   - 模型通过 Transformer 架构处理文本
   
   第三步:生成向量
   - 模型输出固定维度(2048维)的浮点数向量
   - 这个向量捕捉了文本的语义信息
   
   第四步:存储
   - 将向量和原文本一起存储到 VectorStore 中
   - 建立向量索引,便于后续检索
   ```
   
4. **为什么是2048维**:
   ```
   维度选择是精度和性能的权衡:
   
   低维向量(如 128、256 维):
   - 优点:存储空间小、计算速度快、内存占用少
   - 缺点:语义表达能力有限,可能丢失细节信息
   
   高维向量(如 1024、2048、4096 维):
   - 优点:语义表达更精细,能捕捉更多语义细节
   - 缺点:存储和计算成本高,检索速度慢
   
   我们选择2048维的原因:
   - 通义千文 embedding 模型的默认输出维度
   - 在语义精度和性能之间取得良好平衡
   - 对于技术文档这种需要精确语义的场景,2048维足够
   - 内存计算方式存储,2048维不会造成过大压力
   ```

### 3. Transformer 了解吗?原理是什么?

**面试回答要点:**

```
Transformer 是目前 NLP 领域最主流的模型架构,核心创新是 Self-Attention 机制。

核心组件:

1. **Self-Attention(自注意力机制)**:
   - 让模型在处理每个词时,能够关注句子中的所有其他词
   - 计算方式:Q(Query)、K(Key)、V(Value)三个矩阵
   - 公式:Attention(Q,K,V) = softmax(QK^T / d_k) * V
   - 好处:可以捕捉长距离依赖关系,不像 RNN 只能顺序处理

2. **Multi-Head Attention(多头注意力)**:
   - 并行使用多个注意力头,每个头关注不同的语义方面
   - 最后将多个头的输出拼接起来
   - 例如:一个头关注语法关系,一个头关注语义关系

3. **Positional Encoding(位置编码)**:
   - Transformer 本身没有位置信息,需要添加位置编码
   - 让模型知道每个词在序列中的位置

4. **Encoder-Decoder 结构**:
   - Encoder:将输入序列编码为向量表示
   - Decoder:根据编码生成输出序列
   - 我们用的 embedding 模型主要使用 Encoder 部分

为什么用 Transformer 做 embedding:
- 能理解上下文语义,不是简单的词袋模型
- 可以处理长文本,捕捉全局语义
- 生成的向量能准确反映文本的语义相似度
```

### 7. 重排序怎么做?BM25和向量检索的权重比例怎么设定?有什么更好的方案?

**你的项目实现:**

三路召回 + RRF 融合 + LLM Reranker:

```java
// 多路召回器
List<Retriever> retrievers = List.of(
    new SemanticRetriever(vectorStore),      // 向量检索
    new Bm25Retriever(vectorStore),           // BM25关键词检索
    new QueryRewriteRetriever(vectorStore, chatClient) // 查询改写检索
);

// RRF 融合
this.multiRecaller = new MultiRetriever(retrievers);

// LLM Reranker 重排
this.llmReranker = new LlmReranker(baseUrl, apiKey, rerankPath, rerankModel);
```

**面试回答:**

```
**我们采用的是更先进的方案:三路召回 + RRF 融合 + Reranker 重排**

1. **三路召回**:
   - **向量检索(SemanticRetriever)**:
     * 使用余弦相似度
     * 擅长语义匹配,能理解"意思相同但表达不同"的情况
     * 例如:"如何学习Java" 和 "Java入门方法"
   
   - **BM25 关键词检索(Bm25Retriever)**:
     * 计算 TF(词频)和 IDF(逆文档频率)
     * 擅长精确关键词匹配
     * 例如:搜索"Spring Boot"能精准匹配包含这个词的文档
   
   - **查询改写检索(QueryRewriteRetriever)**:
     * 用 LLM 将用户查询改写成多种表达方式
     * 分别检索后合并结果
     * 扩大召回范围,提高召回率

2. **RRF(Reciprocal Rank Fusion)融合算法**:
   - 不需要手动设置权重比例!这是关键优势
   - 公式:RRF(d) = Σ 1 / (k + rank_d)
   - k 是常数(通常取60)
   - 优势:
     * 自动归一化不同检索器的分数
     * 不需要调参
     * 对不同检索器的结果进行合理融合
   
   对比传统加权方案:
```
   传统方案:final_score = 0.7 * bm25_score + 0.3 * vector_score
   问题:权重需要反复调参,不同场景需要不同权重

   RRF方案:基于排名融合,不需要权重参数
   优势:自适应,泛化性好
   ```

3. **LLM Reranker 精确重排**:
   - 使用智谱的专用 Rerank 模型
   - 对 RRF 融合后的候选文档进行精确打分
   - 返回最相关的 top 3 个文档
   - 优势:
     * 专用模型,速度快(比通用 LLM 快 10 倍)
     * 成本低(按调用次数计费,比通用模型便宜)
     * 精度高(专门训练用于相关性排序)
   
4. **降级策略**:
   - 如果 Rerank API 调用失败,回退到 RRF 融合结果
   - 保证系统可用性

**如果面试官问传统方案:**
"传统方案确实是用固定权重,比如 0.2 * 向量得分 + 0.8 * BM25得分,
但这种方案有问题:
- 权重需要大量实验调优
- 不同场景需要不同权重
- 无法适应动态变化

我们的 RRF 方案更先进,不需要设置权重参数,通过排名融合就能达到更好效果。"
   ```

   - Token 消耗减少约 50%

5. **降级策略**:

## 三、对话记忆管理相关问题

### 11. 三层对话记忆压缩策略具体怎么实现的?

**你的项目实现:**

```java
// ChatMemory.java 关键配置
private static final int MAIN_AGENT_MAX_ROUNDS = 20;           // 主Agent最大20轮
private static final int MAIN_AGENT_MAX_ASSISTANT = 3;         // 最多保留3条Assistant回复
private static final int COMPRESS_THRESHOLD_MESSAGES = 60;     // 超过60条触发压缩
private static final int PRESERVE_RECENT_MESSAGES = 20;        // 压缩后保留20条最近消息
```

**面试回答:**

```
**三层记忆管理策略:**

1. **第一层:滑动窗口(基础保护)**
   - 硬性限制最大消息数量(20轮 * 4 = 80条消息)
   - 超过时从头部移除旧消息
   - 特殊处理:不会在 TOOL 消息前截断(保证工具调用完整性)

2. **第二层:Assistant 裁剪(Token优化)**
   - 只保留最近3条 Assistant 回复
   - 原因:LLM 回复通常很长,消耗 token 多
   - 用户问题通常比回复更有价值
   - 早期问题还保留在摘要中,不会丢失信息

3. **第三层:摘要压缩(兜底机制)**
   - 触发条件:历史消息超过60条
   - 保留最近20条消息
   - 将更早的消息通过 LLM 总结为摘要
   - 摘要注入到 System Prompt 中
   
   摘要生成 Prompt:
   - 保留关键信息:用户的核心需求、重要决策、已完成的操作、关键结论
   - 去除冗余:省略寒暄、重复内容、中间推理过程
   - 使用第三人称描述

## 七、算法和数据结构问题

### 16. 手撕算法:图拓扑排序,查看是否有环

**面试回答:**

```java
/**
 * 判断有向图是否有环,如果有环返回[],无环返回拓扑排序结果
 * 使用 DFS + 三色标记法
 */
public List<String> topologicalSort(Map<String, List<String>> graph) {
    // 三色标记:0=未访问,1=访问中,2=已完成
    Map<String, Integer> state = new HashMap<>();
    List<String> result = new ArrayList<>();
    
    // 初始化所有节点状态为0
    for (String node : graph.keySet()) {
        state.put(node, 0);
    }
    
    // 对每个未访问的节点进行DFS
    for (String node : graph.keySet()) {
        if (state.get(node) == 0) {
            if (hasCycle(node, graph, state, result)) {
                return new ArrayList<>(); // 有环返回空列表
            }
        }
    }
    
    Collections.reverse(result); // DFS是逆序的,需要反转
    return result;
}

private boolean hasCycle(String node, Map<String, List<String>> graph, 
                        Map<String, Integer> state, List<String> result) {
    state.put(node, 1); // 标记为访问中
    
    List<String> neighbors = graph.getOrDefault(node, new ArrayList<>());
    for (String neighbor : neighbors) {
        if (state.getOrDefault(neighbor, 0) == 1) {
            return true; // 发现环
        }
        if (state.getOrDefault(neighbor, 0) == 0) {
            if (hasCycle(neighbor, graph, state, result)) {
                return true;
            }
        }
    }
    
    state.put(node, 2); // 标记为已完成
    result.add(node);
    return false;
}

// 测试用例
// A->B B->C C->D        输出:[A,B,C,D]
// A->B B->C C->D D->A   输出:[]
```

**算法说明:**
- 使用 DFS + 三色标记法
- 时间复杂度:O(V+E),V是顶点数,E是边数
- 空间复杂度:O(V),递归栈和状态存储
```
